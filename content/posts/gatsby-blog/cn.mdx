再阅读[这个博客][why-self-host]之后，我决定做一个自己host的博客站点。文中最吸引我的论点是：自己host的站点在未来会提供更多变现的渠道。每个开发者都急需技能变现的渠道，包括我。

在目标确定之后，需要做战术的选择。一般有2种方法制作博客站点：使用专业的博客制作工具如WordPress或从零开始。这两者间的边界有时不是很明确，我将不需要使用者有任何web开发经验的工具分类为专业的博客制作工具。

作为一个半前端开发者，我决定使用一个标准的开发框架：[Gatsby][gatsby]。

# 背景

[Gatsby][gatsby]是一个基于React的多页网站制作框架。它的技术栈如下图所示：

![stack](/image/gatsby-stack.png)

Gatsby的使用者需要了解HTML，CSS，JS，TS和React。因此这也是一个自学前端技能的绝佳机会。

# 挑战

在开始这个项目之前，我对Gatsby没有任何了解。因此我学习Gatsby完成这个项目的过程对任何想学习Gatsby的新手也有参考价值。

## 生成静态网站

作为Next.js开发者，我对服务器渲染（SSR）有一定的了解。但这项技术在Gatsby中不能用，因为Gatsby倾向于生成静态网站（SSG）。这种差异导致我在学习的过程中好几次进入死胡同。下面是SSR和SSG的区别。

| SSR                               | SSG                                        |
|-----------------------------------|--------------------------------------------|
| 允许运行时跑服务器端代码 | 不允许运行时的服务器端代码 |
| 发布时必须搭配服务器程序    | 发布时只能有静态的html/css/...文件 |
| 动态生成网页 | 网页内容变化需要重新编译    |

基于SSG的理念，必须符合下面3个基本要求：

1. 编译时用的代码和运行时用的代码要区分开
2. 要把运行时用的代码当成直接在浏览器中运行的代码来写
3. 保证所有数据都在编译时可以获取

## 国际化

作为base在非英语国家的开发者，我的博客必然要服务于英语和中文的读者。因此我的博客必须支持中英双语。这也是我决定自建博客时的主要考虑因素之一，因为现有的博客平台没有一个在中外都有强大的影响力。

在实现的层面，网络上有很多工具和案例以供参考。

### 使用插件

我尝试的第一个工具是一个Gatsby插件。对非博客站点来说这是最好的方案。我在开发Next.js项目时也用过类似的工具，因此这种方法对我来说不陌生。但是这种方法对博客站点来说并不实用。因为它只支持字段翻译。因其源数据储存在JSON文件中，将其运用在博客站点中会对维护造成极大的不便。没有任何理智的开发者会选择这种方法。

### 自行翻译

基于[官方案例](https://github.com/gatsbyjs/gatsby/tree/master/examples/using-i18n/)，同时支持字段翻译和页面翻译时可行的。

基于以上案例，我制作了一个使用Gatsby，mdx，页面翻译和字段翻译的[模板][template]。

## Markdown和JSX

博客和普通站点最大的区别是博客页面都采用相同的布局。为维护方便，最好将博客的源文件保存在一个过去10年都没有大改的稳定的文件类型中。源文件最好与输出文件非常类似，以此减少overhead，且尽量提供所见即所得的体验。

综合以上考虑，markdown是最佳选择。它非常稳定，即便过去几年有不少增强性改动，但不同的插件间的可交互性非常好，且都保持对旧版本的兼容。

但是原生markdown在功能性上比较欠缺。它的简易性限制了在UI领域的潜能。因此必须进行增强。

JSX在创造好看的UI上非常有用，因此用markdown制作大部分内容，用JSX做细节补全是最佳方案。`gatsby-plugin-mdx`可以满足要求。

在使用mdx的过程中，一些问题需要解决。

### GraphQL

在mdx中，Gatsby允许使用PageQuery。但是我个人不喜欢使用PageQuery。而且它在非页面组件中不能用。因此它在很多场景中的使用很有限。

Sadly, component level query is not available in mdx as the JSX components in mdx are not compiled to retrieve static data during build time. Therefore the codes like `useStaticQuery` is left to run in runtime, where the query will not succeed.
不幸的是，组件级的query在mdx中不可用。因为mdx中的JSX组件在编译时没有获取数据，因此`useStaticQuery`之类的方法被留到运行时才会运行。但运行时query已经不可用了。

### Heading锚

markdown有个很有用的增强，就是heading锚。而且它是TOC的必要前提。但是`gatsby-plugin-mdx`默认不提供这个支持。

基于[此文][heading-anchor]，添加这个特性非常简单。

## 部署

最后一个问题是如何host这个站点。大体上有3种方法：

1. 买个专业hosting服务
2. 在自管服务器上host
3. 找个免费的hosting服务

第一和第二个选项面向懒人，而我找到了一个免费的hosting服务：Github Pages。

根据官方文档，可以轻易地配置Github Pages。要注意的是，Github提供2种Page，项目和个人。适合这个场景的是个人Page。

## Gatsby编译系统

首先要学习的是Gatsby如何编译代码。

大部分源文件都用JSX编写，最终生成一个静态网站。其中大部分问题都已经被Gatsby自带的工具解决了，比如客户端路由。唯一需要使用者额外小心的是数据获取。

有服务器端开发背景的人会下意识地写服务器端运行的代码，但在如Gatsby一样的[SSG工具](#生成静态网站)中这是绝对禁止的。Gatsby仅允许2种运行环境：编译环境和浏览器运行环境。

在编译时，所有数据都要被获取以生成静态HTML文件。在浏览器运行时仅有浏览器API可用。

编译时运行的代码在下列位置：

1. PageQuery和StaticQuery（GraphQL query仅在编译时运行）
2. 项目根目录下的`gatsby-node.js`，`gatsby-config.js`，`gatsby-browser.js`和`gatsby-ssr.js`

其它所有代码都会在编译时变成静态HTML。

想了解如`gatsby-node.js`之类的详细信息，参见[此处][concept].

## GraphQL

Gatsby中的GraphQL在某种意义上是非常规的。在传统的数据获取模型中，客户端先发起请求，后端再根据请求返回数据。我习惯性地认为这个过程发生在运行时。但Gatsby仅在编译时运行所有的GraphQL query，所有的query都不会在输出文件中出现。

因此在编译时必须提供所有需要的数据和schema。一个主要的数据源是`gatsby-config.js`和`gatsby-node.js`。参见[此文][concept]获取详细信息。

## 国际化

为同步实现页面级翻译和字段级翻译，我做了以下设计。

### 页面级

页面级翻译仅支持mdx源文件，例如博客和简历。

一个博客的源文件结构如下图所示：

![post](/i18n-post.png)

所有同一页面的翻译文件都在同一子目录下。子目录的名字就是博客的slug地址。文件名对应locale。所有这些信息都在`gatsby-node.js`中提取并传递至对应的页面。

因为所有页面都有同样的布局，`gatsby-plugin-mdx`可以接收一个模板组件，并基于模板生成所有博客页面。

### 字段级

字段级翻译文件也采用类似的文件结构，文件名对应locale。我制作了一个custom hook用来获取字段。
